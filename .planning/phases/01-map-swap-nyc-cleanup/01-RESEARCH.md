# Phase 1: Map Swap & NYC Cleanup - Research

**Researched:** 2026-02-25
**Domain:** MapLibre GL JS map configuration, Protomaps vector tiles, Buenos Aires geography, NYC data cleanup
**Confidence:** HIGH (codebase analysis) / MEDIUM (Protomaps tile content for BA barrios)

## Summary

Phase 1 replaces the NYC-focused map with Buenos Aires by changing map configuration (center, bounds, zoom) and removing all NYC-specific data and references. The codebase is well-structured with centralized configuration in `src/js/script.js` (the `App.config` object), making the coordinate swap straightforward. NYC references are scattered across JS config, HTML strings, map style JSON files, pipeline code, tag data, and exporter bounding boxes -- all identified via comprehensive grep.

The most significant finding is about barrio labels (MAP-03 requirement). The Protomaps vector tile `places` layer derives neighbourhood labels ONLY from OSM `place=neighbourhood` and `place=suburb` tags (confirmed via Protomaps source code: `Places.java`). Buenos Aires' 48 official barrios are mapped in OpenStreetMap as `boundary=administrative` + `admin_level=9` relations, NOT as `place=neighbourhood` nodes. An Overpass API query confirmed only ~10 informal neighbourhood nodes exist in BA with `place=neighbourhood` (e.g., Barrio Chino, Las Canitas, Abasto) -- the major barrios (Palermo, San Telmo, La Boca, Recoleta) are absent. Therefore, barrio labels will NOT appear automatically from the existing `place-labels-neighborhood` style layer. A custom GeoJSON overlay with a symbol layer is required.

The PIPE-04 requirement (remove NYC data) is straightforward because event/location JSON files are generated by the pipeline exporter. The primary actions are: create empty placeholder data files, replace the `geotags` array in `tags.json` with Buenos Aires barrio names, and replace the `related_tags.json` file with an empty object.

**Primary recommendation:** The map config swap is trivial (change coordinates, bounds, zoom in `App.config`). The barrio labels are the main engineering work -- create a `barrios.geojson` with 48 center points and add a GeoJSON symbol layer to the map, with theme-change resilience.

### Pattern: maxBounds for Map Pan Constraint (MAP-02)

MapLibre's `maxBounds` option constrains panning at the viewport edges. Set during map initialization in the Map constructor:

```javascript
// src/js/script.js, map initialization around line 687
this.state.map = new maplibregl.Map({
    container: 'map',
    style: styleUrl,
    center: [initialView[1], initialView[0]],
    zoom: initialZoom,
    maxZoom: this.config.MAP_MAX_ZOOM,
    maxBounds: [
        [-58.65, -34.78],   // Southwest: [lng, lat] -- generous padding
        [-58.20, -34.48]    // Northeast: [lng, lat] -- generous padding
    ],
    attributionControl: false,
    dragPan: false
});
```

The `maxBounds` should be MORE generous than `CITY_BOUNDS` (which is used for geolocation validation). maxBounds constrains viewport EDGES -- at low zoom the viewport is large, so tight bounds feel restrictive. Use ~0.05-0.10 degrees of extra padding beyond the geolocation bounds.

### Pattern: Custom GeoJSON Barrio Labels (MAP-03)

Since Protomaps tiles will NOT show BA barrio names (see Pitfall 1 below), add a custom GeoJSON source with a symbol layer:

**Step 1:** Create `src/data/barrios.geojson` with 48 barrio center points.

**Step 2:** Add the source and layer programmatically after map load. This must handle theme changes (setStyle destroys custom sources/layers).

Two approaches, in order of preference:

**Approach A (recommended): Programmatic with theme-change resilience**

The existing codebase already handles source/layer recreation after theme changes via `MapManager._ensureLayers()` on the `idle` event. Follow the same pattern for barrio labels -- add the source/layer in the map `load` event, and re-add on `idle` if destroyed by a theme toggle.

```javascript
// After map load, add barrio labels
function addBarrioLabels(map) {
    if (map.getSource('barrios')) return; // Already set up

    fetch('data/barrios.geojson')
        .then(r => r.json())
        .then(geojson => {
            if (map.getSource('barrios')) return;
            map.addSource('barrios', { type: 'geojson', data: geojson });
            map.addLayer({
                id: 'barrio-labels',
                type: 'symbol',
                source: 'barrios',
                minzoom: 11,
                maxzoom: 16,
                layout: {
                    'text-field': ['get', 'name'],
                    'text-font': ['Inter SemiBold'],
                    'text-size': ['interpolate', ['linear'], ['zoom'],
                        11, 10, 13, 12, 15, 14
                    ],
                    'text-transform': 'uppercase',
                    'text-letter-spacing': 0.1,
                    'text-allow-overlap': false,
                    'text-padding': 10
                },
                paint: {
                    'text-color': theme === 'dark' ? '#888' : '#555',
                    'text-halo-color': theme === 'dark' ? '#1a1a1a' : '#f0f0f0',
                    'text-halo-width': 2
                }
            }, 'marker-symbols'); // Insert BEFORE marker layers
        });
}
```

**Approach B: Inline GeoJSON in style JSON files**

Embed the barrio data directly in both `map-style-dark.json` and `map-style-light.json` as a source with inline `data` (not a URL). This avoids the theme-change problem because the style itself carries the data, but it duplicates the barrio data in two files and makes the style files larger.

```json
{
  "sources": {
    "protomaps": { "type": "vector", "url": "..." },
    "barrios": {
      "type": "geojson",
      "data": {
        "type": "FeatureCollection",
        "features": [ ... 48 features ... ]
      }
    }
  }
}
```

**Recommendation:** Use Approach B (inline in style JSON) for simplicity and automatic theme-change resilience. The 48-point GeoJSON is small (~3-4KB) and duplicating it in two style files is negligible. This avoids needing to add theme-change handling code.

### Anti-Patterns to Avoid

- **Don't rewrite the geolocation logic:** Just rename `isWithinNYC` to `isWithinCity` and swap the bounds values. The logic is city-agnostic.
- **Don't delete the tag data files:** Replace their contents. The file paths are referenced in config and must stay.
- **Don't modify the map rendering code (MapManager, ViewportManager):** These are city-agnostic. Only config values change.
- **Don't over-constrain maxBounds:** Make bounds generous. Users should be able to see the full BA metro area including suburbs. Tight bounds feel frustrating, especially at lower zoom levels.
- **Don't add barrio labels as DOM elements:** Use MapLibre's native symbol layer for GPU-accelerated rendering with automatic collision detection.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Map bounds constraining | Custom pan/zoom event handlers | MapLibre `maxBounds` option | Built-in, handles edge cases, works with all zoom levels |
| Barrio label rendering | Custom DOM overlay divs | MapLibre symbol layer with GeoJSON source | GPU-accelerated, handles zoom/collision automatically |
| Barrio center coordinates | Manual lat/lng lookup for each barrio | Extract from OSM Overpass API query | 48 barrios is tedious to manually geocode; Overpass is authoritative |
| Geolocation validation | Custom polygon intersection | Simple bounding box check (existing pattern) | Rectangle is sufficient for city-level validation |

**Key insight:** The barrio GeoJSON file is the one piece of new data that needs careful assembly. Everything else is config value changes.

## Common Pitfalls

### Pitfall 1: Barrio Labels Not Appearing from Protomaps Tiles (CRITICAL)

**What goes wrong:** Assuming the existing `place-labels-neighborhood` style layer will show BA barrio names after recentering the map. The map loads with correct BA geography but shows NO barrio names at any zoom level.
**Why it happens:** Protomaps derives neighbourhood labels ONLY from OSM `place=neighbourhood` and `place=suburb` tags (confirmed in Protomaps `Places.java` source code). Buenos Aires' 48 official barrios are mapped in OSM as `boundary=administrative` + `admin_level=9` relations, which Protomaps ignores for the places layer. Only ~10 informal/unofficial neighbourhood nodes exist (Barrio Chino, Las Canitas, Abasto, etc.) -- the major barrios (Palermo, San Telmo, La Boca, Recoleta) are absent.
**How to avoid:** Create a custom `barrios.geojson` with all 48 official barrio center points and add a dedicated symbol layer.
**Warning signs:** Map loads successfully, streets and parks are visible, but no barrio names appear anywhere.

### Pitfall 2: maxBounds Too Tight

**What goes wrong:** Users cannot see the full BA metro area or the map feels "stuck" when panning near edges.
**Why it happens:** `maxBounds` constrains the map viewport edges, not the center. At lower zoom levels, the viewport is large, so tight bounds prevent seeing the area properly.
**How to avoid:** Set maxBounds ~0.05-0.10 degrees larger than the actual area of interest in each direction. Test at both zoom level 10 (zoomed out) and zoom level 16 (zoomed in).
**Warning signs:** Map snaps back when panning near BA edges; outer barrios like Liniers, Villa Lugano are unreachable.

### Pitfall 3: Coordinate Order Confusion (lat/lng vs lng/lat)

**What goes wrong:** Map centers on wrong location or bounds are inverted.
**Why it happens:** The codebase uses BOTH conventions: `App.config.MAP_INITIAL_VIEW` uses `[lat, lng]` format, while MapLibre's API uses `[lng, lat]`. The map initialization already does the conversion: `center: [initialView[1], initialView[0]]`.
**How to avoid:** Keep `MAP_INITIAL_VIEW` in `[lat, lng]` format (matching the existing convention). Use `[lng, lat]` only for MapLibre API calls like `maxBounds`.
**Warning signs:** Map shows ocean or Africa instead of Buenos Aires.

### Pitfall 4: Missing NYC References in UI Text

**What goes wrong:** After the swap, users see "Fomo NYC" or "NYC events" somewhere in the UI.
**Why it happens:** NYC references are scattered across 15+ files: HTML, JS, PHP, JSON, and comments. Easy to miss one.
**How to avoid:** Use the comprehensive file audit below to address every reference. Run `grep -r "NYC\|New York\|fomo\.nyc" src/ pipeline/ database/` as final verification.
**Warning signs:** User sees mixed branding (BA map with NYC text).

### Pitfall 5: Exporter Bounding Box Still Targets NYC

**What goes wrong:** Running the pipeline export produces empty init JSON files because no events fall within the NYC bounding box.
**Why it happens:** `pipeline/exporter.py` has hardcoded NYC coordinates: `INIT_LAT_RANGE = (40.672945, 40.735535)` and `INIT_LNG_RANGE = (-73.998595, -73.943125)`.
**How to avoid:** Update the exporter bounding box to BA coordinates.
**Warning signs:** `events.init.json` is empty; `events.full.json` has all events; initial page load shows no markers.

### Pitfall 6: related_tags.json is 2MB of NYC Data

**What goes wrong:** The `related_tags.json` file (2MB) is entirely NYC-specific tag associations. Leaving it intact means the tag suggestion system suggests NYC-specific tags like "Brooklyn Art" or "NYC History".
**Why it happens:** This file maps tags to related tags, and ALL entries reference NYC events/locations/categories.
**How to avoid:** Replace contents with empty object `{}`. BA-specific related tags will be generated in Phase 3 when the pipeline processes BA events.
**Warning signs:** Tag suggestions or related tags show "NYC History", "Brooklyn Art", "Queens Events", etc.

### Pitfall 7: Barrio Labels Lost on Theme Toggle

**What goes wrong:** Barrio labels appear initially but disappear when user toggles dark/light theme.
**Why it happens:** Theme toggling calls `map.setStyle()` which destroys ALL custom sources and layers. The existing code handles this for marker layers via `MapManager._ensureLayers()` on the `idle` event, but custom barrio labels would also be destroyed.
**How to avoid:** Either (a) embed barrio GeoJSON inline in both style JSON files (recommended -- simplest), or (b) add barrio source/layer recreation logic to the theme-change handling code path.
**Warning signs:** Toggle theme and barrio labels disappear; toggle back and they're still gone.
